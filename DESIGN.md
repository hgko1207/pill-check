## Problem Statement

60대 부모님이 약국·마트에서 영양제나 일반의약품을 사기 전에, 본인이 복용 중인 처방약과 충돌하는지 즉석에서 확인할 수단이 없다. 약국 직원은 알려주지 않고, 홈쇼핑 광고는 비교 불가능한 정보만 준다. 자녀가 매번 옆에 있을 수도 없다.

**1차 사용자: 부모님 본인 (60대, 디지털 어느 정도 익숙)**. 자녀가 추천해주는 형태가 아니라, 부모님이 직접 손에 들고 매장에서 사용한다.

## What Makes This Cool

**약국이 절대 안 알려주는 정보를, 바코드 한 번에 알려주는 도구.**

비교(가성비)는 인터넷에 이미 있다. 하지만 "와파린 복용 중 + 비타민K 영양제" 또는 "콜레스테롤약(아토르바스타틴) + 종합비타민의 특정 성분" 같은 상호작용 경고는 어디에도 없다. 부모님이 매장에서 마트 셀프계산대 쓰듯이 "삑" 한 번 하면, 빨강/노랑/초록으로 답이 나온다. 그게 "와" 모먼트.

> **참고:** 식약처 DUR(병용금기) 데이터는 처방약 ↔ 처방약 매칭이 주력. 처방약 ↔ 건강기능식품 상호작용 데이터 커버리지는 Week 0 검증 게이트의 핵심 질문. 매칭 실패율이 높으면 즉시 Approach C(LLM 보조) 피벗.

## Constraints

- **솔로 바이브 코딩**. 6개월짜리 풀 프로덕트는 못 만든다. 4~6주말 안에 부모님 손에 들어가야 함.
- **부모님이 직접 사용**. 60대 UX 원칙(18pt+, 한 화면 핵심 3개 이하, 56pt+ 버튼, WCAG AA)이 진짜 진지함.
- **의료 인접 영역**. LLM 환각 책임 회피 필수. 데이터가 없으면 "모름" 솔직하게.
- **잘되면 오픈 가능성**: PWA로 V1부터 양 OS 동시 지원. 공식 오픈 시 PWA URL 그대로 카톡 공유 또는 TWA로 Play Store 진입. App Store 정식은 Capacitor wrap 후 V2.

## Premises (모두 사용자 동의 완료)

1. **바코드 스캔이 1순위**, 제품명 검색이 2순위, OCR은 fallback. 60대가 사진을 정확히 찍는 건 의외로 어렵고, 마트 셀프계산대 경험이 바코드 멘탈 모델을 이미 학습시켜 놓음.
2. **상호작용 경고가 wedge**. 비교는 V2. 솔로 빌더가 4개 기능 다 만들면 출시 못 한다.
3. **자체 백엔드 안 씀**. RN+Node+PostgreSQL은 사이드 프로젝트를 죽인다. **PWA + 클라이언트 IndexedDB + 식약처 API 직접 호출** (CORS 시 Cloudflare Worker proxy 1개만).
4. **식약처 데이터의 한계 인정**. 일반의약품 DUR(병용금기)은 식약처에 있지만, 건강기능식품 ↔ 의약품 상호작용은 데이터 부족. **MVP에서 LLM 안 쓴다** — 책임 회피. 데이터 없으면 "약사 선생님께 확인하세요" 명시.

## Approaches Considered

### Approach A: 동네 약사 (선택됨, **PWA-first**)

**PWA(Progressive Web App)** + 바코드 + 식약처 DUR 직접 매칭. LLM 0. 자체 백엔드 0(필요 시 식약처 CORS proxy 1개만).

- **스택:** Vite + React + TypeScript + html5-qrcode + Dexie(IndexedDB) + 식약처 OpenAPI (의약품안전나라 + 식품안전나라) + Service Worker(오프라인 캐시) + Vercel/Cloudflare Pages 호스팅
- **양 OS 단일 코드베이스:** 부모님(Android Chrome) + 자녀 본인(iOS Safari) 동일 앱. **자녀의 매일 dogfooding이 부모님 베타 전 검증 사이클을 단축.**
- **기능 범위:** ① 부모님 복용 처방약 1~10개 등록(검색 자동완성) → ② 영양제/일반약 바코드 스캔(html5-qrcode) → ③ 식약처 DUR 매칭 → 빨강/노랑/회색/초록 결과.
- **Capacitor fallback 트리거:** Week 0 GATE 또는 부모님 베타에서 PWA 바코드 인식률 < 70%면 Capacitor wrap → 네이티브 ML Kit 카메라로 전환. 코드 90% 재사용.
- **Effort:** CC+gstack ~3~4주말 / 솔로 인간 ~6주 (네이티브 대비 빠름)
- **Completeness:** 7/10 (happy path + 명확한 한계 표시, edge case는 V1.5)
- **공식 오픈 시:** PWA URL 그대로 카톡 공유 = 설치 부담 0. 추후 TWA(Trusted Web Activity)로 Play Store 등록 가능.

### Approach B: 풀 플랫폼 (기각)

Flutter + Supabase + Claude API + 비교 + 가족 공유. 풀 스펙. 2~3개월. **솔로가 못 끝낸다 (90% 확률).**

### Approach C: 약톡 봇 (기각)

채팅 한 화면. 매력적이지만 LLM 비용/지연/검증 부담이 첫 버전으로는 과함. 부모님이 쓰면서 V2에 추가 검토.

## Recommended Approach: A (동네 약사)

부모님 손에 작동하는 앱이 들어가는 시점이 가장 중요한 KPI. A는 그걸 5주 안에 가능하게 한다. 부모님이 매주 쓰시는지 보고, 그 데이터가 B/C로의 진화를 결정한다.

## Data Layer & Sync Strategy

**핵심 결정: 식약처 DUR + 약품 마스터를 1회 일괄 다운로드 후 로컬 저장 (preload, not live).**

이유: 약국·매장에서 LTE/WiFi 불안정 가능. 60대 부모님이 "인터넷 안 돼서 안 됨" 화면 보면 다시 안 엽니다. 검색·매칭은 무조건 오프라인 동작.

**구체적 동작:**
1. **최초 설치 시:** 식약처 의약품안전나라 API에서 일반의약품 + DUR(병용금기) 마스터 일괄 다운로드 → Room에 저장. 진행률 표시 ("약품 정보 받는 중... 12% / 잠시만요"). 예상 데이터 크기 < 50MB 목표.
2. **앱 실행 시:** 저장된 마스터 데이터의 마지막 갱신일 표시 (예: "최근 갱신: 5월 1일"). 30일 이상 된 경우 부드럽게 "업데이트 가능" 배너 (강제 X).
3. **갱신 정책:** 부모님은 직접 갱신 안 해도 OK. 앱 버전 업데이트 시 함께 갱신 (Play Store/TestFlight). V2부터 백그라운드 sync 검토.
4. **오프라인 동작:** 모든 검색·매칭은 오프라인 가능. 앱이 인터넷 없이 약국에서 동작하는 것이 V1 출시 조건.
5. **건강기능식품 마스터:** 식품안전나라 API. 데이터 양이 클 수 있음 — 1차 검증 후 사이즈 결정. 50MB 초과 시 "자주 쓰는 카테고리" subset 다운로드 전략.

**바코드 → 제품 ID 분기 (이슈 surfaces):**
- 한국 의약품 바코드(KD-code, 표준코드) → 식약처 의약품 표준코드 API → 품목기준코드
- 일반 EAN-13(영양제, 일부 OTC) → 식품안전나라 API 또는 제품명 검색 fallback
- ML Kit는 둘 다 읽음. 앱 레벨에서 코드 형식 판별 후 다른 API 분기 호출.
- **Week 0 검증 항목:** 부모님 약 한 개와 흔한 영양제 한 개 둘 다 바코드 → 식약처 ID 매칭 가능한지 확인.

## DUR 색상 매핑 (UI 결정)

식약처 DUR 데이터의 심각도 코드를 4단계 UI 색상에 매핑:

| 색상 | 트리거 조건 | 메시지 |
|---|---|---|
| 🔴 **빨강 (주의)** | DUR 병용금기(절대 금기) 매칭 | "위험 — 함께 드시면 안 됩니다. 약사 선생님께 보여주세요." |
| 🟡 **노랑 (상담 권장)** | DUR 주의(병용주의 또는 용량주의) 매칭 | "주의가 필요합니다. 약사 선생님 상담을 권합니다." |
| ⚪ **회색 (데이터 없음)** | 매칭 데이터 없음 (등록되지 않은 영양제, 처방약 등) | "이 제품의 상호작용 정보가 부족합니다. 약사 선생님께 확인하세요." |
| 🟢 **초록 (확인 안 됨)** | 매칭 결과 명시적으로 안전 (등록된 약 ↔ 등록된 영양제, DUR에 없음) | "현재 등록된 약과의 충돌은 확인되지 않았습니다. (참고용)" |

**중요:** 초록도 "안전 보장"이 아닌 "확인 안 됨" 톤. 모든 결과는 "참고용 — 약사 상담 필수" 면책과 함께 표시.

## 면책 문구 (V1 ship 텍스트)

모든 결과 화면 하단에 고정 표시 (16pt, 회색 배경, 16dp 패딩):

> **본 정보는 식품의약품안전처 공공데이터를 기반으로 한 참고 자료입니다. 의료 행위가 아니며, 약사·의사 상담을 대체하지 않습니다. 실제 복용 결정은 전문가와 상의하세요.**

V2(공식 오픈) 시 법률 검토 후 갱신.

## 처방약 등록 UX (60대 입력 방식)

문제: 60대가 "암로디핀 5mg" 정확히 한글로 타이핑하기 어려움.

**V1 입력 경로 우선순위:**
1. **검색(텍스트) + 자동완성** (1순위): 두 글자 입력 시 식약처 마스터에서 자동완성. "암로" → "암로디핀베실산염정 5mg (노바스크)". 발음 친화 fuzzy match 포함.
2. **약 봉투 사진 OCR** (2순위, V1.5): 처방전 사진 → ML Kit Text Recognition → 약명 추출 → 검색 매칭. 60대 카메라 부담 있어 V1에는 미포함.
3. **음성 입력** (V2): "여보세요, 암로디핀이요" → STT.

**V1 출시 = 검색 자동완성만.** 부모님이 약 이름 일부만 알면 등록 가능해야 함.

**최대 등록 개수:** 10개 (UI 안정성 위해 cap, 실제 60대 평균 복용약 5~7개 커버). 11번째 등록 시 "기존 약 정리 후 추가하세요" 안내.

**개인정보 보호:** 처방약 데이터는 디바이스 로컬 only (Room). 백업 없음. 앱 삭제·디바이스 분실 시 데이터 손실 ← V1에서 명시적으로 trade-off (단순함 우선). V2에서 암호화 백업 검토.

## Open Questions

1. ~~부모님 폰 OS~~ → **확정: 부모님 Android, 자녀 iOS. PWA로 양쪽 동시 지원.**
2. **식약처 OpenAPI 키 발급은 가능한가?** → nedrug.mfds.go.kr / openapi.foodsafetykorea.go.kr 둘 다 키 발급 필요. **수일 소요 가능성** — 이게 critical path. Week 0 즉시 신청, 도착 전까지 모든 빌드 작업 차단.
3. **🚨 DUR API에서 영양제(건강기능식품) ↔ 처방약 매칭이 작동하는가?** → 두 단계 검증:
   - **Week 0 GATE (proof of concept):** 알려진 페어(예: 와파린 ↔ 비타민K) 1건이라도 매칭되면 Week 1 진행. 0건이면 데이터 가용성 자체 부재 — 즉시 Approach C 피벗.
   - **Week 3-4 ship-time 결정:** Week 1-3 동안 빌드 중 다양한 페어 샘플링. 부모님 약 ↔ 흔한 영양제 50개 페어 중 의미있는 매칭률 < 40%면 V1 출시 보류, Approach C 추가 또는 회색 비율 인정 후 출시 결정.
4. **🚨 PWA 바코드 인식률이 충분한가?** → 자녀 iPhone Safari + 부모님 Android Chrome 둘 다에서 흔한 영양제 5개 바코드 5번씩 시도, **각 환경 ≥ 70% 성공이면 PWA 유지**. 한쪽이라도 < 50%면 Week 2~3 중 Capacitor wrap 결정. 50~70%는 회색존 — 부모님 베타 결과 보고 V1.5 결정.
5. **바코드 표준 분기가 실제로 식약처 API와 매칭되는가?** → KD-code(의약품 표준코드)와 EAN-13(영양제) 둘 다 식약처 ID로 변환 가능한지 Week 0에서 함께 검증. 한쪽이 안 되면 fallback 검색 UX로.
6. **CORS 이슈 발생 시 어떻게?** → 식약처 OpenAPI가 브라우저 직접 호출 차단할 가능성. 발생 시 Cloudflare Worker로 1-파일 proxy 작성 (~30분 작업). Week 0에서 같이 확인.
7. **의료 면책 문구 법률 검토?** → 본격 오픈 전. V1은 위 정의된 면책 문구 사용. V2에서 변호사 검토.

## Success Criteria

**V1 출시 = 다음 모두 만족 (측정 가능 기준):**
- [ ] 부모님이 도움 없이 약 등록 1개 → 바코드 스캔 1회 → 결과 확인 완료까지 90초 이내
- [ ] 첫 1주일 내 자발적 사용 ≥ 2회 (앱 실행 로그 기준 — 단, 로컬 카운터만, 외부 텔레메트리 X)
- [ ] V1 출시 후 2주일 추적: 주당 사용 횟수 ≥ 1회 유지

**V1 실패 시그널 (측정 가능):**
- 부모님이 첫 시도에서 약 등록 실패 (검색 fuzzy match 안 됨, 자동완성 인지 못함) → UI 재설계.
- 출시 후 1주일 내 사용 0회 또는 1회 → 가치 제안 검증 실패. 인터뷰 후 C 피벗.
- 매칭률 (스캔 결과 회색 비율) ≥ 60% → 데이터 부족. C(LLM 보조) 검토.

**용어 정의:**
- "V1" = 이 문서 전체에서 "MVP 출시 버전" = 부모님 베타 시작 시점. 향후 LLM 등 도입은 V2로 표기.

## Distribution Plan

**V1 (PWA):** Vercel/Cloudflare Pages 무료 호스팅. 부모님께 카톡으로 URL 한 번 → "홈 화면에 추가" 안내(첫 진입 시 큰 안내 카드 노출). 가족만 비밀 URL 공유.
- 자녀 본인 iPhone: 동일 URL, 매일 dogfooding
- 부모님 Android: 동일 URL, 홈 화면 아이콘으로 설치된 앱처럼 사용

**V1.5 (Capacitor 트리거 시):** PWA 코드 그대로 + Capacitor wrap → APK 빌드 → Android 부모님 폰 sideload. 자녀 iOS는 Xcode 빌드 또는 PWA 유지.

**V2 (공식 오픈 시):**
- Option A: PWA URL 그대로 공개 + TWA(Trusted Web Activity)로 Play Store 등록
- Option B: Capacitor APK + IPA로 양 스토어 정식 등록 + 의료 면책 법률 검토 + 개인정보처리방침
- 그 시점 트래픽·피드백에 따라 선택

## Tech Stack (확정 — PWA-first)

## What I noticed about how you think

- **"다 좋은데 자녀가 골라주는건 아니고 부모님이 쓸거야"** — 이 한 문장이 세션 통틀어 가장 중요한 결정이었습니다. "자녀 추천 모드"라는 흔한 패턴을 단호하게 거부했고, 그게 가족 공유 기능 우선순위를 즉시 낮춰줬습니다. 사용자 정의에 대한 명확한 그림이 있다는 신호.
- **"잘되면 공식 오픈도 생각있어"** — 이걸 짊어진 채로 사이드 프로젝트로 시작하는 건 흔한 함정인데(완벽주의로 넘어가서 출시 못 함), 동시에 4개 전제 도전을 모두 받아들이고 가장 빠른 길을 택했습니다. 야망과 현실주의가 같이 있는 드문 조합.
- **"추천이 뭐야? 동네 약사야?"** — 결정을 빠르게 정리하는 스타일. 시간 낭비 없는 빌더 페이스. 다만 그래서 더, 매주 부모님 사용 데이터를 정직하게 마주하는 것이 중요해집니다 — 빠른 페이스는 잘못된 가설을 빠르게 굳히기도 쉬워요.
